## 分布式训练探索

### 探索历程

| 阶段 | 方案 | 精度 | 结论 |
|------|------|:---:|------|
| 1 | Spark MLlib RF/GBT | 52.1% | 分桶近似信息损失大，**放弃** |
| 2 | 纯numpy手写RF (单机) | 86.3% | 算法=sklearn，精度差0.2%，**保留** |
| 3 | numpy RF 广播+map (Spark) | 5.7% | 模型容量不足 |
| 4 | 全链路手动MapReduce (Spark) | 待定 | 特征工程+树训练全分布 |

### MLlib 为何精度低

```
sklearn/numpy:  每个分裂点精确计算 ──→ 最优分裂 ──→ 86.5%
MLlib:          连续特征离散化为maxBins个桶 ──→ 近似分裂 ──→ 52.1%
```

MLlib的`maxBins=32`将10K个连续值映射到32个桶，丢失了细粒度分裂信息。在30K行小数据上信息损失远大于大数据场景。

### numpy手写RF vs sklearn

| | sklearn RF | numpy手写RF |
|------|:---:|:---:|
| 分裂算法 | CART精确分裂 | CART精确分裂 |
| 实现 | Cython编译 | Python+numpy |
| 精度(设备级) | 86.5% | 86.3% |
| 速度 | 4秒 | 283秒 (71×慢) |
| 并行 | 自动多核 | 单线程 |
| 分布式 | ✗ | ✓ 可分发树到Worker |

**关键发现**: 算法层面手写RF和sklearn完全等价（精确分裂+Bootstrap+Bagging）。71倍速度差来自Python解释执行 vs Cython编译。分布式训练可弥补速度劣势。

### Spark全链路分布式架构

```
┌──────────────────────────────────────────────────────────────┐
│ Phase 1 [Driver]  加载CSV → 拟合LabelEncoder → 预计算时间特征  │
│                   29,964行 → list[dict] → 广播Encoder          │
├──────────────────────────────────────────────────────────────┤
│ Phase 2 [Map]     parallelize(raw_rows, 24分区)               │
│                   每行 → (hash(device_id)%24, encoded_row)     │
├──────────────────────────────────────────────────────────────┤
│ Phase 3 [Shuffle] partitionBy(24)                            │
│                   ★ 手动实现: 同设备路由到同一Worker            │
├──────────────────────────────────────────────────────────────┤
│ Phase 4 [Reduce]  mapPartitions: sort→group→compute_features  │
│                   ★ 手动reduceByKey: 分区内排序+相邻合并       │
│                   → 13,339个事件行                             │
├──────────────────────────────────────────────────────────────┤
│ Phase 5 [Driver]  collect → 过滤 → 划分 → StandardScaler      │
│                   10,654 train / 2,685 test                    │
├──────────────────────────────────────────────────────────────┤
│ Phase 6 [Scout]   broadcast(X_train) → parallelize(100)       │
│                   100树×depth=10 → 特征重要性 → Top10           │
├──────────────────────────────────────────────────────────────┤
│ Phase 7 [Main]    broadcast(X_train[:,top10]) → parallelize(300)│
│                   300树×depth=18 → 并行Bootstrap训练            │
├──────────────────────────────────────────────────────────────┤
│ Phase 8 [Driver]  collect(300棵树) → 集成预测 → 评估 → 保存    │
└──────────────────────────────────────────────────────────────┘
```

**手动实现的Spark核心算子：**

| 算子 | 实现方式 |
|------|---------|
| `HashPartitioner` | `md5(device_id) % 24` → 决定数据分区 |
| `Shuffle` | `partitionBy(N)` → 跨Worker网络重分布 |
| `reduceByKey`/`groupByKey` | 分区内`sort(device_id)` → 相邻同key合并 |
| `map` | 每棵树独立Bootstrap+训练 |

### 分布式训练Bug排查记录

| Bug | 位置 | 表现 | 修复 |
|-----|------|------|------|
| **响应时间算错** | `_compute_device_events` | `response = (fault_ts - prev_ts)` = 故障间隔, 不是响应时间 | 改为 `response = (repair_start_ts - fault_ts)` |
| **时区陷阱** | `_compute_device_events` | `datetime.fromtimestamp()` 在CentOS(UTC+0)把UTC+8数据的时间特征(hour/weekday/month)全算错 | Driver端预计算时间特征, Worker直接用 |
| **hist追加NaN** | `_compute_device_events` | 无效间隔追加NaN → hist混入NaN → `np.nanmean`与`np.mean`不一致 | 无效间隔直接跳过, 不追加 |

## 关键决策记录

1. **设备级 > 事件级**: 聚合同一设备多次故障取平均MTBF，比单次间隔预测稳定得多（86.9% vs 72.0%）
2. **不用MLlib**: 分桶近似分裂点在小数据上精度损失太大（52%），改用手写精确分裂
3. **numpy RF = sklearn (算法层面)**: 71×慢于sklearn源于Python解释执行 vs Cython编译
4. **分布式意义的边界**: 1MB训练数据，通信开销极低，真正收益在于树训练的并行化
5. **LabelEncoder vs hash编码**: LabelEncoder保留类别有序性，精度比hash高
6. **事件级精度天花板**: 单次故障间隔随机性157%，物理上限约72%
7. **Driver端预计算时间特征**: 避免Worker上`datetime.fromtimestamp()`的跨OS时区陷阱
