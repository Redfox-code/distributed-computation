# 交通大数据与分布式计算 — 2026最终作业 操作总结

---

## 项目概览

基于AFC设备维修日志数据，构建设备故障间隔时间(MTBF)回归预测模型，支撑地铁AFC设备预防性维修调度决策。

**最终精度：**

| 方案 | 粒度 | 1-MAPE | 框架 |
|------|------|:------:|------|
| sklearn RF | 设备级 | **86.5%** | sklearn Driver |
| sklearn RF | 事件级 | **71.0%** | Spark + sklearn |

**核心技术路线：** 分布式MapReduce特征工程（Spark） → 建模数据集 → sklearn RF训练（Driver） → 评估预测

## 3. Task2: 分布式事件级特征工程

### 3.1 分布式架构

```
                   Driver (Master001)
                        │
         ┌──────────────┼──────────────┐
         ▼              ▼              ▼
    Slave001        Slave002        Slave003
   (4核/768M)      (4核/768M)      (4核/768M)

Phase 1 [Driver]:  加载CSV → 预计算时间特征 → 广播LabelEncoder
Phase 2 [Map]:     parallelize(29,964行, 24分区)
                   每行 → (md5(device)%24, encoded_row)
Phase 3 [Shuffle]: partitionBy(24) — 同device路由到同一Worker
Phase 4 [Reduce]:  mapPartitions → 分区内排序 → 相邻同key合并
                   → 每设备严格时序计算24个衍生特征
Phase 5 [Driver]:  collect → Pearson|r| + 互信息 + VIF 三重筛选
                   → 分期划分 + SMOTE + RF200验证
```

### 3.2 手动实现的MapReduce算子

| 算子 | 实现 |
|------|------|
| `HashPartitioner` | `md5(device_id) % 24` → 分区分配 |
| `Shuffle` | `partitionBy(24)` → 跨Worker网络重分布 |
| `reduceByKey` | 分区内 `sort(device_id)` → 相邻同key合并 |
| `map` | 每设备独立计算24个衍生特征（严格时序防泄漏） |

### 3.3 衍生特征（24个）

**时间特征(5):** 故障小时、故障星期、故障月份、故障季度、是否周末、是否工作日高峰
**维修行为(5):** 维修时长_小时、响应时间_小时、维修类型、平均维修时长_小时、响应时间趋势
**历史统计(12):** 历史次数、上次间隔_小时、平均间隔_小时、间隔标准差_小时、近期趋势_比值、最近3次平均_小时、最近3次趋势、运行天数、故障频率_次每天、故障加速比
**类别编码(3):** 车站编码、品牌编码、线路编码

### 3.4 三重特征筛选

```
24个衍生特征
    │
    ├─ Pearson |r| ≥ 0.03 ──────────→ 15个通过
    │   (筛选出: 故障频率_次每天 |r|=0.31, 历史次数 |r|=0.27, ...)
    │   (剔除: 维修时长_小时 |r|=0.007, 响应时间_小时 |r|=0.004)
    │
    ├─ 互信息 MI ≥ 0.001 ──────────→ 验证非线性关系
    │
    └─ VIF ≤ 10 ───────────────────→ 12个通过
        (剔除: 故障月份 VIF=107.8, 平均间隔_小时 VIF=42.0)
```

**关键发现：** 全部24特征输出给Task3，筛选仅用于验证报告。Task3的`SelectFromModel`动态选Top10比静态筛选更有效（70.9% vs 66.8%）。

## 4. Task3: 模型构建

### 4.1 设备级回归（86.5%）

每台设备一行，目标 = 平均MTBF，样本量 ~1,523台。

| 模型 | MAE | MAPE | 1-MAPE | R² |
|------|:---:|:---:|:------:|:---:|
| **RF** | 3.8天 | 13.5% | **86.5%** | 0.867 |

**关键处理：** log1p目标变换 + 按MTBF分层划分 + StandardScaler

### 4.2 事件级回归（71.0%）

每次故障一行，目标 = 下次故障间隔，样本量 ~13,339次。

**最终架构（task3_event_spark_full.py）：**

```
Phase 1 [Driver]: 读取Task2建模数据集 → 13原版特征
Phase 2 [Driver]: 按设备80/20划分 → StandardScaler → log1p变换
Phase 3 [分布式]: sc.broadcast(X,y) → parallelize(100) → mapPartitions
                  → 100棵scout NumPyTree训练 → collect → 算特征重要性 → Top10
Phase 4 [Driver]: sklearn RF(300树×depth20×leaf2×split5) 训练
Phase 5 [Driver]: 预测 → expm1 → 评估
```

**为什么Phase 4用sklearn而不是NumPyTree：**

| 方案 | 1-MAPE | 原因 |
|------|:------:|------|
| NumPyTree (本地Windows) | 70.5% | 算法正确 ✓ |
| NumPyTree (Spark Worker) | -195.9% | **pickle序列化回Driver时树结构损坏** |
| sklearn (Driver) | **71.0%** | Cython编译, 无序列化问题 |

经过多轮诊断确认：NumPyTree在Worker上训练正常（40K+字符的dict树结构），但通过`collect()`传回Driver时pickle反序列化导致predict方法返回错误值。这是PySpark对象序列化的已知限制——复杂嵌套Python对象的跨进程传输不可靠。

**解决方案：Phase 3仍用NumPyTree（仅需feature importance，数组序列化正常），Phase 4用sklearn。**

### 4.3 NumPyTree调试关键发现

| Bug | 位置 | 症状 | 修复 |
|-----|------|------|------|
| **range少1** | `_best_split` | `range(ml, n-ml)` → 最小叶子=3而非2 | `range(ml-1, n-ml)` |
| **Worker序列化** | `collect()` | 树dict损坏, predict返回噪声 | 改用sklearn Driver |
| response算错 | 分布式FE | `(fault_ts-prev_ts)` = 故障间隔 ≠ 响应时间 | `(repair_start_ts-fault_ts)` |
| 时区陷阱 | 分布式FE | `datetime.fromtimestamp()` 时区错 | Driver预计算时间特征 |

## 6. 关键决策记录

1. **设备级 > 事件级**: 均值消除单次随机性（86.5% vs 71.0%）
2. **不用MLlib**: 分桶近似精度损失大（52%），改用精确分裂
3. **不用NumPyTree做最终训练**: PySpark pickle序列化复杂对象不可靠
4. **分布式在特征工程不在树训练**: 1MB数据无需分布式训练，MapReduce特征工程才是Spark的正确用法
5. **全部特征 > 静态筛选**: Task3的SelectFromModel动态选Top10优于Task2静态筛选
6. **事件级天花板 ~72%**: 单次故障间隔CV=122%，物理上限
